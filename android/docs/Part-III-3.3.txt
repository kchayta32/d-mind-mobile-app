3.3 การออกแบบการทำงานของซอฟต์แวร์
	การออกแบบการทำงานของซอฟต์แวร์ สำหรับแอปพลิเคชันระบบเฝ้าระวังและแจ้งเตือนภัยพิบัติอัจฉริยะ ซึ่งเป็นระบบติดตามภัยพิบัติและจัดการเหตุการณ์ภาวะฉุกเฉินบนอุปกรณ์เคลื่อนที่ระบบปฏิบัติการแอนดรอยด์แบบเนทีฟที่พัฒนาด้วยภาษาคอตลินร่วมกับ Jetpack Compose และ Material 3 สำหรับสร้างส่วนติดต่อผู้ใช้ โดยมี Java native services สำหรับรองรับ Firebase Cloud Messaging ทำงานเบื้องหลังเพื่อติดตามตำแหน่ง การตรวจสอบพื้นที่เสี่ยงแบบ Geofence และระบบแจ้งเตือนของแอนดรอยด์ ส่วนโค้ด TypeScript React Vite และ Capacitor เดิมใช้เป็นต้นแบบและแหล่งอ้างอิงของเว็บแอปพลิเคชันเดิมเท่านั้น 
โดยโปรแกรมจะรับข้อมูลจากทั้งฝั่งฮาร์ดแวร์และข้อมูลจากแหล่งภายนอกที่เปิดให้บริการฟรี เช่น ข้อมูลแผ่นดินไหวจากสำนักสำรวจธรณีวิทยาสหรัฐอเมริกา ข้อมูลจุดความร้อนจากสำนักงานพัฒนาเทคโนโลยีอวกาศและภูมิสารสนเทศ และข้อมูลพยากรณ์อากาศจากกรมอุตุนิยมวิทยา ซึ่งมีรายละเอียดซอฟต์แวร์ ดังต่อไปนี้
3.3.1 สถาปัตยกรรมโดยรวมของระบบ
ระบบซอฟต์แวร์ของแอปพลิเคชันระบบเฝ้าระวังและแจ้งเตือนภัยพิบัติอัจฉริยะ ได้รับการออกแบบเป็นสถาปัตยกรรมแบบ 5 ชั้น โดยประยุกต์แนวคิดการแบ่งชั้นของแบบจำลองโอเอสไอเพื่อแยกหน้าที่ของส่วนต่อประสานผู้ใช้ การประมวลผล การให้บริการ การจัดการข้อมูล และการสื่อสารกับอุปกรณ์หรือบริการภายนอกออกจากกันอย่างชัดเจน มีรายละเอียดมีดังนี้
3.3.1.1 ชั้นส่วนติดต่อผู้ใช้ (Presentation Layer) 
ในส่วนโปรแกรมประยุกต์แอนดรอยด์แบบเนทีฟ พัฒนาด้วย Jetpack Compose บนภาษา Kotlin โดยมี MainActivity.kt เป็นจุดเริ่มต้นและเรียกใช้ DMindApp.kt เป็นโครงสร้างหลักของส่วนติดต่อผู้ใช้ แบ่งการทำงานออกเป็นหน้าจอตามหน้าที่ของระบบ
ในกลุ่ม ui.screens และใช้ Material 3 เป็นแนวทางในการออกแบบส่วนติดต่อผู้ใช้เพื่อให้รูปแบบการแสดงผลมีความสอดคล้อง ใช้งานง่ายและเหมาะสมกับอุปกรณ์เคลื่อนที่
3.3.1.2 ชั้นตรรกะทางธุรกิจ (Business Logic Layer)
ใช้ ViewModel สำหรับจัดการสถานะและประสานการทำงานระหว่างส่วนติดต่อผู้ใช้กับตรรกะของระบบ ใช้ยูสเคสเป็นขั้นตอนการเรียกใช้ข้อมูลตามขอบเขตงาน เช่น ข้อมูลภัยพิบัติ การค้นหาสถานที่ จุดความร้อน น้ำท่วม และชั้นแผนที่ของ GISTDA ใช้ Repository เป็นตัวกลางในการเข้าถึงข้อมูล และใช้ Kotlin coroutines สำหรับการประมวลผลแบบไม่ประสานเวลา


3.3.1.3 ชั้นบริการ (Service Layer)
ใช้ส่วนหลังที่พัฒนาด้วย Ktor เป็นชั้นบริการกลางสำหรับคำขอที่ต้องผ่านแม่ข่าย เช่น การตรวจสอบสถานะระบบ การดึงข้อมูลการแจ้งเตือน การรับรายงานเหตุการณ์ การดึงข้อมูลพยากรณ์อากาศและการส่งการแจ้งเตือน โดยช่วยแยกตรรกะฝั่งแม่ข่ายออกจากตรรกะฝั่งโปรแกรมประยุกต์
3.3.1.4 ชั้นข้อมูล (Data Layer)
ทำหน้าที่ดึงข้อมูล บันทึกข้อมูล และจัดการหน่วยความจำแคช (Cache) ของโปรแกรมประยุกต์อย่างเป็นระบบ ประกอบด้วย
1) แหล่งข้อมูลระยะไกล (Remote Data Source) ใช้ไฟล์ SupabaseRestClient.kt และไฟล์ SupabaseRepository.kt สำหรับเชื่อมต่อกับฐานข้อมูลหลักซูปาเบสและเรียกใช้งานเอจฟังก์ชันผ่านเครือข่าย HTTP โดยพัฒนาระบบรับบริการตามสถาปัตยกรรมเรสต์ (REST Client) ขึ้นมาใช้เองโดยอาศัยคลาส HttpURLConnection เพื่อลดการพึ่งพาไลบรารีภายนอกและชุดพัฒนาซอฟต์แวร์ (SDK) ของซูปาเบส
2) แหล่งข้อมูลท้องถิ่น (Local Data Source) ใช้ระบบจัดการฐานข้อมูลรูม (Room Database) ผ่านไฟล์ DMindRoomDatabase.kt และระบบฐานข้อมูลเอสคิวไลต์ (SQLite Database) ผ่าน AlertsCacheDAO.java ในการทำหน้าที่จัดเก็บหน่วยความจำแคชการแจ้งเตือน ประวัติตำแหน่งที่ตั้งของผู้ใช้งานและข้อมูลพิกัดพื้นที่อันตราย เพื่อให้ระบบมีความสามารถในการทำงานแบบออฟไลน์ในภาวะเครือข่ายไม่เสถียร
3.3.1.5 ชั้นการสื่อสารและการเชื่อมต่อภายนอก (Communication and External Integration Layer)
ทำหน้าที่เชื่อมต่อกับแหล่งข้อมูลและบริการภายนอก เช่น GISTDA สำหรับข้อมูลจุดความร้อน น้ำท่วม ค่าฝุ่น PM2.5 และชั้นแผนที่ TMD สำหรับข้อมูลพยากรณ์อากาศ ThaiLLM หรือ Supabase Edge Function สำหรับผู้ช่วยเอไอ และ Firebase Cloud Messaging สำหรับการแจ้งเตือนแบบพุช โดยชั้นนี้ช่วยแยกรายละเอียดของการสื่อสาร เครือข่ายและรูปแบบข้อมูลภายนอกออกจากตรรกะหลักของระบบ

3.3.3 การออกแบบแผนที่ภัยพิบัติ
แผนที่ภัยพิบัติเป็นองค์ประกอบหลักของแอปพลิเคชัน ใช้ MapLibre Android SDK สำหรับแสดงแผนที่แบบโต้ตอบได้ภายในหน้าจอที่พัฒนาด้วย Jetpack Compose ฐานแผนที่หลักใช้ไทล์ภาพราสเตอร์ของ OpenStreetMap ผ่านรูปแบบ URL ไทล์และสามารถเลือกฐานแผนที่อื่นที่รองรับในแอป เช่น แผนที่ภูมิประเทศและภาพถ่ายดาวเทียม
3.3.3.1 ฐานแผนที่และการแสดงผล
	ระบบกำหนดรูปแบบแผนที่ด้วยโครงสร้างแบบ MapLibre โดยเพิ่มแหล่งข้อมูลแบบ raster source สำหรับฐานแผนที่และเพิ่มชั้นข้อมูลภัยพิบัติจาก GISTDA เป็นชั้นข้อมูลซ้อนทับเมื่อผู้ใช้เลือกประเภทภัยพิบัติที่ต้องการตรวจสอบ การแสดงผลจุดหรือพื้นที่บนแผนที่รองรับการปักหมุด การจัดกลุ่มข้อมูลจำนวนมากและการแสดงรายละเอียดในแผงข้อมูลด้านล่าง
3.3.3.2 ชั้นข้อมูลภัยพิบัติและแหล่งข้อมูล
ระบบแผนที่ของแอปพลิเคชันรองรับข้อมูลภัยพิบัติและข้อมูลประกอบจากหลายแหล่ง ได้แก่



1) ข้อมูลจาก GISTDA
1.1) ข้อมูลจุดความร้อนจาก GISTDA VIIRS สำหรับแสดงตำแหน่งจุดความร้อนและไฟป่า
1.2) ข้อมูลพื้นที่น้ำท่วมจาก GISTDA flood สำหรับแสดงพื้นที่น้ำท่วมล่าสุด
1.3) ข้อมูลน้ำท่วมซ้ำซากจาก GISTDA flood-freq 
เพื่อประเมินประวัติการเกิดอุทกภัยซ้ำในอดีต
1.4) ข้อมูลพื้นที่และปัญหาการแพร่กระจายของผักตบชวาในแหล่งน้ำธรรมชาติจาก GISTDA water_hyacinth
1.5) ได้แก่ ดัชนีภัยแล้ง (DRIPlus) ดัชนีความเปียกชื้นของพืชพรรณ (NDWI) และดัชนีความชื้นในชั้นดินระดับตื้น (SMAP) เพื่อวิเคราะห์ความแห้งแล้งสะสม
2) ข้อมูลแผ่นดินไหวจาก USGS สำหรับแสดงรายละเอียดทางธรณีวิทยา มีขนาดเป็นแมกนิจูด (Mw)
3) ข้อมูลสภาพอากาศจาก TMD สำหรับใช้ในการประเมินและ
แจ้งเตือนจากพยากรณ์อากาศล่วงหน้าเชิงภูมิภาคและรายพิกัดตำแหน่งที่ตั้ง
4) ข้อมูลโครงข่ายอุตุนิยมวิทยาประเทศไทยจาก Open-Meteo
5) ข้อมูลฐานแผนที่และการค้นหาสถานที่จาก OpenStreetMap และ Nominatim สำหรับแสดงแผนที่พื้นฐานและค้นหาตำแหน่งภายในประเทศไทย
3.3.3.3 การค้นหาและการกรองข้อมูลบนแผนที่
ผู้ใช้งานสามารถจัดการตัวเลือกบนหน้าจอแผนที่ผ่านแดชบอร์ด 
ซึ่งประกอบด้วยส่วนควบคุมแสดงผลแบบแบ่งกลุ่มสำหรับสลับระหว่างโหมดแสดงข้อมูลภัยพิบัติกับโหมดแสดงข้อมูลสถานีตรวจวัดดี-มายด์ แผงปุ่มฟังก์ชันควบคุมแผนที่ ปุ่มระบุตำแหน่งที่ตั้ง ปุ่มสลับชั้นข้อมูลและระบบสืบค้นข้อมูลผ่านแถบค้นหาที่เชื่อมต่อกับเอพีไอของ Nominatim ผู้ใช้งานสามารถเลือกกรองชั้นข้อมูลภัยพิบัติแยกตามประเภท พร้อมเลือกช่วงเวลาของการประเมิน เช่น 1 วัน, 3 วัน, 7 วัน หรือ 30 วัน นอกจากนี้ยังออกแบบแผงข้อมูลแสดงผลแบบเลื่อนขึ้นจากด้านล่าง (Bottom Sheet) สำหรับสรุปรายงานสถานการณ์และตัวชี้วัดสถิติต่าง ๆ
3.3.3.4 ข้อมูลจากสถานีตรวจวัด
	ระบบใช้สถานีตรวจวัดเพื่อเก็บข้อมูลสภาพแวดล้อมภาคสนามประกอบด้วยอุปกรณ์ตรวจวัด GY-521 BME280 AJ-SR04T และ PMS5003 สำหรับตรวจวัดค่าที่เกี่ยวข้องกับระดับน้ำ สภาพอากาศ การเอียงของพื้นดินและฝุ่นละออง ข้อมูลจากอุปกรณ์ตรวจวัดถูกส่งผ่าน MQTT ไปยังราสเบอร์รี่พายจากนั้นประมวลผลและจัดทำเป็น
เอพีไอด้วย FastAPI เพื่อให้แอปพลิเคชันนำไปใช้แสดงผล


 

ภาพที่ 3.5 หน้าจอแสดงแผนที่ภัยพิบัติ

3.3.4 การออกแบบระบบพยากรณ์อากาศ
ระบบพยากรณ์อากาศดึงข้อมูลจาก 2 แหล่งหลัก ดังนี้
3.3.4.1 ข้อมูลจากกรมอุตุนิยมวิทยา
1) สำหรับพยากรณ์ตามพิกัดละติจูดและลองจิจูดของผู้ใช้
2) สำหรับพยากรณ์ตามภูมิภาคของประเทศไทย
3) แสดงข้อมูลอุณหภูมิ ความชื้น ความกดอากาศ ปริมาณฝน ความเร็วลม ปริมาณเมฆ และสภาพอากาศทั่วไป
		3.3.4.2 ข้อมูลสภาพอากาศและคุณภาพอากาศจาก Open-Meteo
1) สำหรับพยากรณ์ข้อมูลสภาพอากาศแบบรายชั่วโมงและรายวันประกอบการวิเคราะห์
2) ดึงข้อมูลค่าฝุ่นละอองขนาดเล็ก (PM2.5) และดัชนีคุณภาพอากาศ (AQI) เพื่อเฝ้าระวังสิ่งแวดล้อม
3) แสดงค่าอุณหภูมิเสมือนจริง ความดันบรรยากาศระดับน้ำทะเล และทิศทางลมประกอบการพยากรณ์

3.3.10 การออกแบบระบบย่อยเพื่อรองรับการขยายบริการในอนาคต
เพื่อให้ระบบมีความสามารถในการขยายตัว (Scalability) และเอื้อประโยชน์ต่อการทดสอบความน่าเชื่อถือโดยไม่เกิดค่าใช้จ่ายและผลกระทบด้านความปลอดภัยของข้อมูลจริง ระบบจึงได้พัฒนาสถาปัตยกรรมระบบย่อยรองรับแผนการให้บริการในอนาคต (Future Service Expansion) ไว้ดังนี้
3.3.10.1 ระบบบันทึกข้อมูลและติดตามสถานะผู้ประสบภัย (Victim Tracking System)
ระบบได้เตรียมส่วนอินเตอร์เฟสการกรอกข้อมูลและโครงข่ายการส่ง payload ไปยังตารางข้อมูลหลัก แต่กำหนดสิทธิ์การเชื่อมต่อ (Access Control) ให้ยอมรับเฉพาะชุดข้อมูลทดสอบภายใน และกำหนดสิทธิ์การอ่านแบบ Real-time Websocket Gateway ไว้เป็นสถานะปิดล็อกชั่วคราวเพื่อรอการเปิดใช้ร่วมกับหน่วยบรรเทาสาธารณภัยจริง
3.3.10.2 ระบบประเมินความเสียหายเชิงกายภาพ (Damage Assessment System)
ออกแบบโครงสร้างการอัปโหลดไฟล์ภาพสิ่งปลูกสร้างที่เสียหายเข้าสู่ Supabase Storage Bucket ที่เตรียมไว้ แต่ใช้การจำลองผลลัพธ์ผ่านตัวตอบกลับเชิงสถิต (Static Response Placeholders) และจำกัดการประเมินวิเคราะห์จาก AI ในระดับ Production เพื่อควบคุมค่าใช้จ่ายในการเรียกใช้งาน Edge Function
3.3.10.3 ระบบวิเคราะห์ข้อมูลและสรุปแนวโน้มสิ่งแวดล้อม (Disaster & Environmental Analytics)
สำหรับฟังก์ชันวิเคราะห์สถิติภัยพิบัติและสิ่งแวดล้อม ได้มีการสร้างโมเดลจำลอง API Endpoint ดึงสถิติย้อนหลังไว้ครบถ้วน แต่ล็อกการทำงานโดยจำกัดการเรียกใช้งานชุดคีย์ API ของ GISTDA และ TMD ปริมาณสูงในช่วงทดสอบ โดยเปลี่ยนเป็นการเรียกใช้ข้อมูลประวัติท้องถิ่น (Local historical fallbacks) แทนเพื่อรักษาสิทธิ์ปริมาณการใช้งาน API
