3.3 การออกแบบการทำงานของซอฟต์แวร์
	การออกแบบการทำงานของซอฟต์แวร์ สำหรับแอปพลิเคชันระบบเฝ้าระวังและแจ้งเตือนภัยพิบัติอัจฉริยะ ซึ่งเป็นระบบติดตามภัยพิบัติและจัดการเหตุการณ์ภาวะฉุกเฉินบนอุปกรณ์เคลื่อนที่ระบบปฏิบัติการแอนดรอยด์แบบเนทีฟที่พัฒนาด้วยภาษาคอตลินร่วมกับ Jetpack Compose และ Material 3 สำหรับสร้างส่วนติดต่อผู้ใช้ โดยมี Java native services สำหรับรองรับ Firebase Cloud Messaging ทำงานเบื้องหลังเพื่อติดตามตำแหน่ง การตรวจสอบพื้นที่เสี่ยงแบบ Geofence และระบบแจ้งเตือนของแอนดรอยด์ ส่วนโค้ด TypeScript React Vite และ Capacitor เดิมใช้เป็นต้นแบบและแหล่งอ้างอิงของเว็บแอปพลิเคชันเดิมเท่านั้น 
โดยโปรแกรมจะรับข้อมูลจากทั้งฝั่งฮาร์ดแวร์และข้อมูลจากแหล่งภายนอกที่เปิดให้บริการฟรี เช่น ข้อมูลแผ่นดินไหวจากสำนักสำรวจธรณีวิทยาสหรัฐอเมริกา ข้อมูลจุดความร้อนจากสำนักงานพัฒนาเทคโนโลยีอวกาศและภูมิสารสนเทศ และข้อมูลพยากรณ์อากาศจากกรมอุตุนิยมวิทยา ซึ่งมีรายละเอียดซอฟต์แวร์ ดังต่อไปนี้
3.3.1 สถาปัตยกรรมโดยรวมของระบบ
ระบบซอฟต์แวร์ของแอปพลิเคชันระบบเฝ้าระวังและแจ้งเตือนภัยพิบัติอัจฉริยะ ได้รับการออกแบบเป็นสถาปัตยกรรมแบบ 5 ชั้น โดยประยุกต์แนวคิดการแบ่งชั้นของแบบจำลองโอเอสไอเพื่อแยกหน้าที่ของส่วนต่อประสานผู้ใช้ การประมวลผล การให้บริการ การจัดการข้อมูล และการสื่อสารกับอุปกรณ์หรือบริการภายนอกออกจากกันอย่างชัดเจน มีรายละเอียดมีดังนี้
3.3.1.1 ชั้นส่วนติดต่อผู้ใช้ (Presentation Layer) 
ในส่วนโปรแกรมประยุกต์แอนดรอยด์แบบเนทีฟ พัฒนาด้วย Jetpack Compose บนภาษา Kotlin โดยมี MainActivity.kt เป็นจุดเริ่มต้นและเรียกใช้ DMindApp.kt เป็นโครงสร้างหลักของส่วนติดต่อผู้ใช้ แบ่งการทำงานออกเป็นหน้าจอตามหน้าที่ของระบบ
ในกลุ่ม ui.screens และใช้ Material 3 เป็นแนวทางในการออกแบบส่วนติดต่อผู้ใช้เพื่อให้รูปแบบการแสดงผลมีความสอดคล้อง ใช้งานง่ายและเหมาะสมกับอุปกรณ์เคลื่อนที่
3.3.1.2 ชั้นตรรกะทางธุรกิจ (Business Logic Layer)
	ทำหน้าที่จัดการสถานะและประมวลผลตรรกะการทำงานของโปรแกรมประยุกต์อย่างเป็นระบบ ประกอบด้วย
	1) วิวโมเดล (ViewModel) มีหน้าที่จัดการข้อมูลและรักษาสถานะการแสดงผลของหน้าจอแต่ละส่วน โดยแยกการทำงานออกจากวงจรชีวิต (Lifecycle) ของแอนดรอยด์ ส่วนประกอบ (Component) ที่สำคัญ เช่น DisasterMapViewModel (จัดการสถานะแผนที่และตัวเลือกชั้นข้อมูล), ChatbotViewModel (จัดการการสนทนากับแบบจำลอง Dr.Mind), ReportViewModel (จัดการสถานะการส่งรายงานเหตุการณ์), DamageAssessmentViewModel (จัดการวิเคราะห์รูปภาพเพื่อประเมินความเสียหาย), และ AnalyticsDashboardViewModel (ประมวลผลสถิติภาพรวม)
	2) กรณีการใช้งาน (Use Case) ทำหน้าที่ควบคุมขั้นตอนตรรกะการดำเนินงานของระบบแยกตามขอบเขตงานเพื่อความยืดหยุ่น เช่น GetDisasterSnapshotUseCase, GetViirsHotspotsUseCase, GetFloodFeaturesUseCase, FetchWeatherForCoordsUseCase, และ SearchPlacesUseCase
	3) คลังข้อมูล (Repository) ทำหน้าที่ประสานและทำตัวเชื่อมกลางในการเข้าถึงข้อมูลจากหลายแหล่ง ประกอบไปด้วย SupabaseRepository (จัดการข้อมูลหลัก), DisasterRepository (จัดการข้อมูลแผนที่และการค้นหาตำแหน่งที่ตั้ง), และ NativeStatusRepository ซึ่งจัดการคุณสมบัติเฉพาะของอุปกรณ์ระบบปฏิบัติการแอนดรอยด์ เช่น บริการติดตามตำแหน่งที่ตั้งในส่วนหลัง (Background Monitoring), การตรวจสอบสิทธิ์การเข้าถึง และการตรวจจับพื้นที่เสี่ยงด้วยขอบเขตภูมิศาสตร์จำลอง (Geofence), การเพิ่มประสิทธิภาพการใช้พลังงานแบตเตอรี่ (Battery Optimization), และการข้ามโหมดห้ามรบกวน (Do Not Disturb - DND Bypass)
	4) คอรูทีน (Coroutines) และสายข้อมูลโฟลว์ (Kotlin Flow) ใช้สำหรับประมวลผลและส่งกระแสข้อมูลแบบไม่ประสานเวลา (Asynchronous Content Streaming)


3.3.1.3 ชั้นบริการ (Service Layer)
ใช้ส่วนหลังที่พัฒนาด้วย Ktor เป็นชั้นบริการกลางสำหรับคำขอที่ต้องผ่านแม่ข่าย เช่น การตรวจสอบสถานะระบบ การดึงข้อมูลการแจ้งเตือน การรับรายงานเหตุการณ์ การดึงข้อมูลพยากรณ์อากาศและการส่งการแจ้งเตือน โดยช่วยแยกตรรกะฝั่งแม่ข่ายออกจากตรรกะฝั่งโปรแกรมประยุกต์
3.3.1.4 ชั้นข้อมูล (Data Layer)
	ทำหน้าที่ดึงข้อมูล บันทึกข้อมูล และจัดการหน่วยความจำแคช (Cache) ของโปรแกรมประยุกต์อย่างเป็นระบบ ประกอบด้วย
	1) แหล่งข้อมูลระยะไกล (Remote Data Source) ใช้ SupabaseRestClient และ SupabaseRepository สำหรับเชื่อมต่อกับฐานข้อมูลหลักซูปาเบส และเรียกใช้งานฟังก์ชันประมวลผลส่วนขอบเครือข่าย (Edge Functions) ผ่านเครือข่าย HTTP โดยพัฒนาระบบรับบริการตามสถาปัตยกรรมเรสต์ (REST Client) ขึ้นมาใช้เองโดยอาศัยคลาส HttpURLConnection ในระดับโปรแกรมประยุกต์ เพื่อลดการพึ่งพาห้องสมุดโปรแกรมภายนอกและชุดพัฒนาซอฟต์แวร์ (SDK) ของซูปาเบส นอกเหนือจากนี้ยังมี BackendRestClient สำหรับเชื่อมต่อกับบริการส่วนหลังที่พัฒนาด้วย Ktor (Ktor Backend) เพื่อลงทะเบียนรับรหัสประจำเครื่องรับบริการส่งข้อมูลผ่านคลาวด์ (FCM Device Token) และส่งรายงานข้อมูลการช่วยเหลือฉุกเฉิน
	2) แหล่งข้อมูลท้องถิ่น (Local Data Source) ใช้ระบบจัดการฐานข้อมูลรูม (Room Database) ผ่านไฟล์ DMindRoomDatabase.kt และระบบฐานข้อมูลเอสคิวไลต์ (SQLite Database) ดั้งเดิมผ่าน AlertsCacheDAO.java ในการทำหน้าที่จัดเก็บหน่วยความจำแคชการแจ้งเตือน (Alerts Cache), ประวัติตำแหน่งที่ตั้งของผู้ใช้งาน (Location History), คิวข้อความการขอความช่วยเหลือฉุกเฉินตกค้าง (SOS Queue) และข้อมูลพิกัดพื้นที่อันตราย (Danger Zones) เพื่อให้ระบบมีความสามารถในการทำงานแบบออฟไลน์เป็นอันดับแรก (Offline-First Capability) แม้ในภาวะเครือข่ายไม่เสถียร
3.3.1.5 ชั้นการสื่อสารและการเชื่อมต่อภายนอก (Communication and External Integration Layer)
ทำหน้าที่เชื่อมต่อกับแหล่งข้อมูลและบริการภายนอก เช่น GISTDA สำหรับข้อมูลจุดความร้อน น้ำท่วม ค่าฝุ่น PM2.5 และชั้นแผนที่ TMD สำหรับข้อมูลพยากรณ์อากาศ ThaiLLM หรือ Supabase Edge Function สำหรับผู้ช่วยเอไอ และ Firebase Cloud Messaging สำหรับการแจ้งเตือนแบบพุช โดยชั้นนี้ช่วยแยกรายละเอียดของการสื่อสาร เครือข่ายและรูปแบบข้อมูลภายนอกออกจากตรรกะหลักของระบบ

3.3.3 การออกแบบแผนที่ภัยพิบัติ
แผนที่ภัยพิบัติเป็นองค์ประกอบหลักของแอปพลิเคชัน ใช้ MapLibre Android SDK สำหรับแสดงแผนที่แบบโต้ตอบได้ภายในหน้าจอที่พัฒนาด้วย Jetpack Compose ฐานแผนที่หลักใช้ไทล์ภาพราสเตอร์ของ OpenStreetMap ผ่านรูปแบบ URL ไทล์และสามารถเลือกฐานแผนที่อื่นที่รองรับในแอป เช่น แผนที่ภูมิประเทศและภาพถ่ายดาวเทียม
3.3.3.1 ฐานแผนที่และการแสดงผล
	ระบบกำหนดรูปแบบแผนที่ด้วยโครงสร้างแบบ MapLibre โดยเพิ่มแหล่งข้อมูลแบบ raster source สำหรับฐานแผนที่และเพิ่มชั้นข้อมูลภัยพิบัติจาก GISTDA เป็นชั้นข้อมูลซ้อนทับเมื่อผู้ใช้เลือกประเภทภัยพิบัติที่ต้องการตรวจสอบ การแสดงผลจุดหรือพื้นที่บนแผนที่รองรับการปักหมุด การจัดกลุ่มข้อมูลจำนวนมากและการแสดงรายละเอียดในแผงข้อมูลด้านล่าง
3.3.3.2 ชั้นข้อมูลภัยพิบัติและแหล่งข้อมูล
ระบบแผนที่ของโปรแกรมประยุกต์รองรับข้อมูลภัยพิบัติและข้อมูลวิเคราะห์ประกอบเชิงลึกจากหลายแหล่ง ได้แก่
1) ข้อมูลภูมิสารสนเทศจากสำนักงานพัฒนาเทคโนโลยีอวกาศและภูมิสารสนเทศ (GISTDA)
   1.1) ข้อมูลจุดความร้อนจาก GISTDA VIIRS (Feature Resource) ดึงข้อมูลย้อนหลัง 1 วัน สำหรับตรวจสอบพิกัดจุดความร้อนและเฝ้าระวังไฟป่า
   1.2) ข้อมูลพื้นที่น้ำท่วมจาก GISTDA flood (Feature และชั้นข้อมูลซ้อนทับ WMS) แสดงพื้นที่น้ำท่วมล่าสุด
   1.3) ข้อมูลพื้นที่น้ำท่วมซ้ำซากจาก GISTDA flood-freq (ชั้นข้อมูลซ้อนทับ WMS) เพื่อประเมินประวัติการเกิดอุทกภัยซ้ำในอดีต
   1.4) ข้อมูลพื้นที่และปัญหาการแพร่กระจายของผักตบชวาในแหล่งน้ำธรรมชาติจาก GISTDA water_hyacinth ผ่านส่วนต่อประสานโปรแกรมประยุกต์ (API)
   1.5) ข้อมูลดัชนีวิเคราะห์ภัยแล้งและความชื้นผ่านบริการแผนที่ผ่านเว็บ (Web Map Service - WMS) ได้แก่ ดัชนีภัยแล้ง (DRIPlus), ดัชนีความเปียกชื้นของพืชพรรณ (NDWI) และดัชนีความชื้นในชั้นดินระดับตื้น (SMAP) เพื่อวิเคราะห์ความแห้งแล้งสะสม
2) ข้อมูลแผ่นดินไหวทั่วโลกจากสำนักงานสำรวจธรณีวิทยาสหรัฐอเมริกา (USGS) แสดงรายละเอียดทางธรณีวิทยา ขนาด (Magnitude) และความลึกแบบรายสัปดาห์
3) ข้อมูลสภาพอากาศและพยากรณ์อากาศจากกรมอุตุนิยมวิทยา (TMD) สำหรับใช้ในการประเมินและแจ้งเตือนภัยล่วงหน้าเชิงภูมิภาคและรายพิกัดตำแหน่งที่ตั้ง
4) ข้อมูลโครงข่ายอุตุนิยมวิทยาและการไหลของน้ำรอบประเทศไทยจากส่วนต่อประสานโปรแกรมประยุกต์ Open-Meteo
   4.1) ข้อมูลอัตราการระบายน้ำของแม่น้ำและแนวโน้มอุทกภัย (River Discharge) จาก Open-Meteo Flood API
   4.2) ข้อมูลความหนาแน่นและความร้อนชื้นสะสมหน้าดิน (Soil Moisture Heatmap) จาก Open-Meteo Land/Forestry API
5) ข้อมูลบริการแผนที่ฐาน (Base Map Tiles) และระบบแปลงรหัสที่อยู่เป็นพิกัดทางภูมิศาสตร์ (Geocoding Service) จาก OpenStreetMap และ Nominatim สำหรับวิเคราะห์ค้นหาตำแหน่งภายในประเทศไทย

3.3.3.3 การค้นหาและการกรองข้อมูลบนแผนที่
ผู้ใช้งานสามารถจัดการตัวเลือกบนหน้าจอแผนที่ผ่านแผงควบคุมการดำเนินงาน (Dashboard) ซึ่งประกอบด้วยส่วนควบคุมแสดงผลแบบแบ่งกลุ่ม (Segmented Controls) สำหรับสลับระหว่างโหมดแสดงข้อมูลภัยพิบัติกับโหมดแสดงเครือข่ายสถานีตรวจวัด D-MIND, แผงปุ่มฟังก์ชันควบคุมแผนที่ (ปุ่มขยายเข้า, ปุ่มขยายออก, ปุ่มระบุตำแหน่งที่ตั้ง และปุ่มสลับชั้นข้อมูล) และระบบสืบค้นข้อมูลผ่านแถบค้นหา (Search Bar) ที่ประสานงานกับส่วนต่อประสานโปรแกรมประยุกต์ Nominatim ผู้ใช้งานสามารถเลือกกรองชั้นข้อมูลภัยพิบัติแยกตามประเภท พร้อมเลือกช่วงเวลาของการประเมิน เช่น 1 วัน, 3 วัน, 7 วัน หรือ 30 วัน ระบบยังออกแบบแผงข้อมูลแสดงผลเลื่อนขึ้นด้านล่าง (Bottom Sheet) แบบยืดหยุ่นที่ด้านล่างสำหรับสรุปรายงานสถานการณ์ การ์ดตัวชี้วัดสถิติ และการแสดงแผนภูมิแนวโน้มสถิติภัยพิบัติอย่างละเอียดผ่านห้องสมุดโปรแกรม Vico Compose M3

3.3.3.4 ข้อมูลจากสถานีตรวจวัด
	ระบบรองรับเครือข่ายสถานีตรวจวัด D-MIND ภาคสนามเพื่อเก็บข้อมูลสภาพแวดล้อมจริงผ่านแผงวงจรตรวจวัด (Sensor Module) ประกอบด้วย เซ็นเซอร์ตรวจจับมุมเอียงทางกลศาสตร์ GY-521 (ป้องภัยดินถล่ม), เซ็นเซอร์ตรวจวัดสภาพแวดล้อมทางอุตุนิยมวิทยา BME280 (อุณหภูมิ ความชื้น ความดันอากาศ), เซ็นเซอร์ตรวจวัดระดับน้ำด้วยคลื่นความถี่เหนือเสียง AJ-SR04T (เฝ้าระวังน้ำท่วมเฉียบพลัน) และเซ็นเซอร์ตรวจวัดความหนาแน่นฝุ่นละอองเชิงแสง PMS5003 (ตรวจจับฝุ่นละออง PM2.5 และ PM10) ข้อมูลดังกล่าวถูกส่งผ่านโปรโตคอลคิวส่งข้อความสำหรับอุปกรณ์ตรวจวัดระยะไกล (MQTT Protocol) ไปประมวลผลส่วนขอบ (Edge Computing) บนเครื่องราสเบอร์รี่พาย จากนั้นจะนำขึ้นฐานข้อมูลเพื่อเรียกใช้งานผ่านบริการเว็บเอพีไอที่พัฒนาด้วยกรอบการทำงาน FastAPI (FastAPI Framework) ในการแสดงผลสถานะการทำงาน (ออนไลน์, แจ้งเตือน หรือออฟไลน์) บนโปรแกรมประยุกต์

3.3.4 การออกแบบระบบพยากรณ์อากาศ
ระบบพยากรณ์อากาศดึงข้อมูลจาก 2 แหล่งหลัก ดังนี้
3.3.4.1 ข้อมูลจากกรมอุตุนิยมวิทยา
1) สำหรับพยากรณ์ตามพิกัดละติจูดและลองจิจูดของผู้ใช้
2) สำหรับพยากรณ์ตามภูมิภาคของประเทศไทย
3) แสดงข้อมูลอุณหภูมิ ความชื้น ความกดอากาศ ปริมาณฝน ความเร็วลม ปริมาณเมฆ และสภาพอากาศทั่วไป
3.3.4.2 ข้อมูลสภาพอากาศและคุณภาพอากาศจาก Open-Meteo
1) สำหรับพยากรณ์ข้อมูลสภาพอากาศแบบรายชั่วโมงและรายวันประกอบการวิเคราะห์
2) ดึงข้อมูลค่าฝุ่นละอองขนาดเล็ก (PM2.5) และดัชนีคุณภาพอากาศ (AQI) เพื่อเฝ้าระวังสิ่งแวดล้อม
3) แสดงค่าอุณหภูมิเสมือนจริง (Apparent Temperature) ความดันบรรยากาศระดับน้ำทะเล (Sea Level Pressure) และทิศทางลมประกอบการพยากรณ์

3.3.6 การออกแบบระบบแจ้งเตือน
ระบบแจ้งเตือนของโปรแกรมประยุกต์ได้รับการพัฒนาตามโครงสร้างพื้นฐานระบบแจ้งเตือนแบบส่งตรงของแอนดรอยด์ (Native Android Push Infrastructure) ทำหน้าที่ส่งต่อสัญญาณแจ้งเตือนภัยพิบัติไปยังเครื่องรับบริการของผู้ใช้ปลายทาง โดยระบบแบ่งกลไกการทำงานออกเป็นสองระดับหลักดังนี้:
1) การส่งข้อมูลด้วยบริการรับส่งข้อความผ่านคลาวด์ของไฟร์เบส (Firebase Cloud Messaging - FCM): โปรแกรมประยุกต์จะสื่อสารกับชุดพัฒนาซอฟต์แวร์ FCM (FCM SDK) และจัดส่งรหัสเครื่องรับบริการ (FCM Device Token) ไปลงทะเบียนยังเซิร์ฟเวอร์ส่วนหลังที่พัฒนาด้วย Ktor (Ktor Backend) ที่พิกัด `/fcm/register` เมื่อเครื่องแม่ข่ายยิงคำขอแจ้งเตือนภัยตามขอบเขตพื้นที่ที่ระบุ บริการ FCM Service ภายในเครื่องผู้ใช้ (FCMFirebaseService.java) จะรับภาระข้อมูลที่เป็นประโยชน์ (Data Payload) อันประกอบไปด้วยรหัสหลัก เช่น alert_type, alert_title, และ alert_message เพื่อประมวลผลและเรียกใช้งานตัวควบคุม EmergencyNotificationManager ต่อไป
2) การกำหนดช่องทางการแจ้งเตือนของระบบปฏิบัติการแอนดรอยด์ (Android Notification Channels): เพื่อให้เป็นไปตามมาตรฐานการควบคุมระบบความปลอดภัยและสิทธิ์ขอการแจ้งเตือน (POST_NOTIFICATIONS) ของระบบปฏิบัติการแอนดรอยด์ เวอร์ชัน 13 ขึ้นไป ระบบได้ออกแบบและสร้างช่องทางสัญญาณแจ้งเตือนแยกตามการปฏิบัติงานหลักสามส่วนดังแสดงในตารางที่ 3.2

ตารางที่ 3.2 ระบบแบ่งช่องทางการแจ้งเตือนเป็น 3 ระดับ
ช่องทางการแจ้งเตือน	ระดับความสำคัญการแจ้งเตือน	รายละเอียดการนำเสนอ
ฉุกเฉิน (emergency_alerts) 	สูงสุด (IMPORTANCE_HIGH)	ส่งแจ้งภัยพิบัติรุนแรงฉุกเฉินในทันที พร้อมเสียงสัญญาณเฉพาะ ระบบสั่นสะเทือน รองรับการข้ามโหมดห้ามรบกวน (DND Bypass) และการเรียกเปิดกิจกรรมการเตือนแบบเต็มหน้าจอ (Full-screen intent ผ่าน EmergencyAlertActivity)
สำคัญ (sos_messages) 	สูง (IMPORTANCE_HIGH)	ใช้สำหรับการแจ้งเตือนและติดตามสถานะกระบวนการขอความช่วยเหลือกรณีฉุกเฉิน (SOS Status Update) ทำงานร่วมกับเสียงและสัญญาณสั่นตามปกติ
ทั่วไป (background_operations) 	ปกติ/ต่ำ (IMPORTANCE_DEFAULT)	ใช้สำหรับแสดงการทำงานของบริการในส่วนหลัง เช่น แสดงแถบการติดตามพิกัดตำแหน่งที่ตั้งของบริการ BackgroundLocationService ในรูปแบบการแจ้งเตือนแบบติดค้างคงรูป (Persistent Notification) เพื่อป้องกันไม่ให้ระบบปฏิบัติการยกเลิกกระบวนการทำงานเบื้องหลัง

3.3.7 การกำหนดช่องทางการแจ้งเตือนตามระดับความรุนแรง
ระบบประยุกต์ใช้ขั้นตอนวิธีจับคู่ระดับความรุนแรง (Severity Mapping Algorithm) อัตโนมัติ โดยอ้างอิงระดับความรุนแรงของสถานการณ์ภัยพิบัติ ตั้งแต่ระดับ 1 (เฝ้าระวังทั่วไป) จนถึงระดับ 5 (วิกฤตความเสียหายสูงสุด) โดยระบบประเมินผลระดับ 4-5 จะถูกส่งไปยังช่องทางแจ้งเตือนระดับฉุกเฉินสูงสุด ระดับ 3 ส่งไปยังช่องทางสำคัญ และระดับ 1-2 ส่งไปยังช่องทางทั่วไปอย่างเงียบ โดยผ่านขั้นตอนประเมินในระดับสถาปัตยกรรมดังภาพที่ 3.9

ภาพที่ 3.9 แผนภาพการทำงานของระบบแจ้งเตือน

3.3.8 การออกแบบชั้นข้อมูลและฐานข้อมูล
3.3.8.1 บริการส่วนหลังของซูปาเบส (Supabase Backend Service)
ระบบประยุกต์สถาปัตยกรรมบริการส่วนหลังของ Supabase เป็นแกนกลางในการรักษาสถานะ ซิงค์ข้อมูล และบริการแบบจำลองปัญญาประดิษฐ์ (AI) จากเว็บแอปพลิเคชันเดิม ประกอบด้วยส่วนประกอบสำคัญสามส่วน:
1) ผังโครงสร้างฐานข้อมูลโพสต์เกรส (PostgreSQL Database Schema) ออกแบบตารางในฐานข้อมูลเชิงสัมพันธ์เพื่อจัดเก็บข้อมูลสอดคล้องกันทั่วทั้งระบบ ประกอบไปด้วย:
   - incident_reports: เก็บบันทึกข้อมูลรายงานเหตุการณ์ภัยพิบัติจากผู้ใช้งานทั่วไป รวมถึงระบุพิกัดตำแหน่งที่ตั้งและที่อยู่รูปภาพหลักฐานเหตุการณ์
   - victim_reports: จัดเก็บข้อมูลประวัติและสถานะขอความช่วยเหลือเร่งด่วนของผู้ประสบภัย
   - realtime_alerts: บันทึกพิกัด พื้นที่ และประเภทภัยธรรมชาติที่มีผลในปัจจุบันเพื่อประเมินความเสี่ยงร่วมกับขอบเขตภูมิศาสตร์จำลอง (Geofence)
   - user_notification_settings: บันทึกการกำหนดค่าและตัวเลือกการเปิดรับข้อมูลแจ้งเตือนของผู้ใช้งานแต่ละเครื่อง
   - damage_assessments: จัดเก็บข้อมูลผลวิเคราะห์ประเมินระดับความเสียหายของทรัพย์สินที่ได้รับการประเมินด้วยปัญญาประดิษฐ์
   - satisfaction_surveys: บันทึกคำแนะนำและผลคะแนนประเมินความพึงพอใจการใช้งานระบบ
2) บริการประมวลผลฟังก์ชันส่วนขอบเครือข่าย (Edge Functions Service) ใช้พัฒนาฟังก์ชันตรรกะแบบกระจายศูนย์เพื่อลดระยะเวลาประมวลผล:
   - ai-chat: ฟังก์ชัน Edge สำหรับควบคุมกระบวนการสนทนาระหว่างเครื่องรับบริการกับแบบจำลองภาษาขนาดใหญ่สำหรับภาษาไทย ThaiLLM (Typhoon Model) เพื่อประมวลผลคำตอบและการปฏิบัติตนที่ถูกต้องล่วงหน้า
   - analyze-damage: ฟังก์ชัน Edge สำหรับรับข้อมูลรูปภาพความเสียหายและพิกัดภูมิศาสตร์ จากนั้นจะสั่งงานตัวแบบคอมพิวเตอร์วิทัศน์ (Computer Vision Model) เพื่อระบุระดับความรุนแรงและวิเคราะห์ความเสียหายของโครงสร้างอาคารเพื่อนำไปใช้วางแผนภารกิจกู้ภัย
3) การจัดเก็บข้อมูลสื่อสิ่งประดิษฐ์ภายนอก (Supabase Storage Buckets) รองรับการจัดเก็บรูปภาพพยานและรูปภาพประเมินทรัพย์สิน โดยระบบจะทำรหัสเชื่อมโยงภาพแบบเข้าถึงภายนอกเชิงสาธารณะ (Public URL) ผ่านระบบลูกข่าย REST Client ทันทีหลังสิ้นสุดการอัปโหลดไฟล์
3.3.8.2 ส่วนต่อประสานโปรแกรมประยุกต์ (API) จากแหล่งภายนอก
	ระบบประยุกต์ใช้งานส่วนต่อประสานโปรแกรมประยุกต์และบริการภายนอกร่วมกับเครือข่ายฐานข้อมูลภัยพิบัติ โดยดำเนินการดึงข้อมูลและมีรอบวงจรการล้างและปรับปรุงข้อมูลแคช (Cache Refresh Lifecycle) ทุก ๆ 5 นาที ผ่านคลังข้อมูล (Repository) ในโปรแกรมประยุกต์ รายละเอียดบริการหลักและประเภทข้อมูลสรุปได้ดังตารางที่ 3.3
